# Architecture

## System Overview

Odeon is a three-process system. Each process has a single clear responsibility. They never directly share memory or state.

```
┌─────────────────────────────────────────────┐
│  apps/desktop (Tauri)                        │
│                                              │
│  ┌──────────────────────────────────────┐    │
│  │  React + TypeScript + Tailwind + Zustand │ │
│  │  DAW-style UI                        │    │
│  └────────────────┬─────────────────────┘    │
│                   │ Tauri invoke / events     │
│  ┌────────────────▼─────────────────────┐    │
│  │  Rust core (src-tauri)               │    │
│  │  tauri-plugin-shell sidecar bridge   │    │
│  └────────────────┬─────────────────────┘    │
└───────────────────┼─────────────────────────-┘
                    │ JSON-RPC over stdio
                    ▼
┌───────────────────────────────────┐
│  apps/audio-engine (C++ binary)   │
│  Tracktion Engine + JUCE           │
│  Headless console sidecar          │
│  AudioEngineBridge implementation  │
└───────────────────────────────────┘

apps/desktop ──HTTP localhost:8000──► apps/api (FastAPI + SQLite)
```

## Why This Separation

### Playback is native (C++)

Web Audio API cannot deliver the sample-accurate synchronization, stable clocking, sub-millisecond latency, and offline render quality required for professional DAW-style playback. All real audio processing goes through Tracktion Engine.

The engine is a headless console app. It does not know about React, HTTP, or the UI. It only knows about JSON-RPC messages and WAV files.

### Python is offline analysis only

Python (librosa, pyloudnorm, scipy) is excellent for audio analysis but cannot drive real-time audio. The FastAPI service analyzes files after they land on disk, updates SQLite, and returns structured JSON. It never touches real-time audio playback.

### React is UI only

React renders the DAW interface and manages application state (Zustand). It calls the FastAPI service over HTTP for data and calls the engine via Tauri invoke commands. It does not contain any audio processing logic.

## Communication Protocols

### Desktop ↔ Engine (JSON-RPC over stdio)

The engine binary runs as a Tauri sidecar. The Rust layer spawns it and owns its stdin/stdout. All bridge commands are encoded as line-delimited JSON objects.

Request:
```json
{"id": 1, "method": "createTrack", "params": {"trackId": "...", "name": "My Drums", "role": "user_stem", "stemType": "drums"}}
```

Response:
```json
{"id": 1, "result": {"ok": true, "result": "..."}}
```

Async events (no id):
```json
{"event": "transportState", "isPlaying": true, "positionSeconds": 4.21, "bpm": 120.0}
{"event": "trackMeters", "meters": {"track-1": {"leftDb": -12.3, "rightDb": -13.1}}}
```

### Desktop ↔ API (HTTP/JSON)

Standard REST. The React frontend calls the FastAPI service at `http://localhost:8000`. The API returns full `OdeonProject` objects after each mutation. All types are defined in `packages/shared` (TypeScript) and mirrored as Pydantic models in the API.

### Shared Data

The `audio/` directory is the shared data store:
- Uploads live at `audio/uploads/{project_id}/`
- Separated stems at `audio/stems/{project_id}/`
- Renders at `audio/renders/`
- SQLite DB at `audio/odeon.db`

Both the API and the engine read/write absolute file paths. The desktop passes file paths it gets from the API to the engine.

## Project Data Flow

```
User uploads reference WAV
          │
          ▼
FastAPI: save to audio/uploads/{project_id}/
         create OdeonTrack (reference_full_mix)
         analyze_track() → TrackAnalysis
         attempt stem separation (Demucs if available)
         if stems: create OdeonTrack per stem (reference_stem)
         return OdeonProject
          │
          ▼
React: update Zustand projectStore
       render tracks in TrackList
       useEngineSync: for each new track, call engine createTrack + addClip
          │
          ▼
Engine: load WAV into Tracktion Edit
        track is ready to play sample-synced
```

## Key Design Rules

1. **No audio in React.** React cannot call Web Audio for real mixing. Only use the engine.
2. **No playback in Python.** FastAPI only reads WAV files for analysis. No audio output.
3. **No DB in the engine.** The engine only knows file paths and play/stop/seek commands.
4. **Types are shared.** `packages/shared` defines all interfaces. The API Pydantic models mirror them exactly.
5. **Evidence required.** Every MixMove generated by the API includes evidence, confidence, and daw_ready_parameters. No fake precision.
